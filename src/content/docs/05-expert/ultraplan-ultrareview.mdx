## Concept — 4 phases, 4 critères de sortie

UltraPlan/UltraReview découpe un travail ambitieux en **4 phases séquentielles**, chacune avec son rôle, son budget tokens, et son **critère de sortie explicite** avant de passer à la suivante.

```
[1 Plan]   → plan.md             ← critère : plan complet, sans ambiguïtés
   ↓
[2 Review] → revised-plan.md     ← critère : plan challengé et défendu
   ↓
[3 Implement] → code + tests     ← critère : tests verts, plan suivi
   ↓
[4 Judge] → verdict.md           ← critère : score ≥ 7/10, concerns listés
```

Si un critère n'est pas atteint, on **ne passe pas** à la phase suivante — on itère dans la phase courante ou on revient en arrière.

## Phase 1 — UltraPlan

**Objectif** : produire un plan d'implémentation **détaillé et défendable**, pas un document vague.

Structure attendue de `plan.md` :

```markdown
# Plan — <nom de la tâche>

## Objectif (1 phrase)
## Contraintes et non-buts
## Fichiers à toucher (avec raison)
## Étapes ordonnées (avec dépendances)
## Risques identifiés
## Critère de succès (comment on sait que c'est fini)
```

Subagent dédié :

```markdown
---
name: ultra-planner
description: Produit un plan d'implémentation complet pour une tâche non triviale. Interroge jusqu'à éliminer les ambiguïtés.
tools: Read, Grep, Glob
model: opus
---
Tu es planner. Tu NE CODES PAS. Tu NE DÉCIDES PAS à la place de l'utilisateur.
Étapes :
1. Lis les fichiers mentionnés + contextes adjacents.
2. Liste les questions qui te bloquent. Si > 0, demande.
3. Propose un plan structuré selon le template.
Budget : 10 tours max. Si tu dépasses, la tâche est trop grosse — propose un split.
```

**Critère de sortie** : le plan peut être lu par quelqu'un qui n'a pas vu la conversation, et il/elle peut exécuter. Pas de "figure out X" ou "as needed".

## Phase 2 — UltraReview

**Objectif** : challenger le plan. Chercher les failles, pas valider.

Subagent distinct (important) :

```markdown
---
name: ultra-reviewer
description: Review critique d'un plan. Cherche failles, cas manquants, dérives silencieuses.
tools: Read, Grep, Glob
model: opus
---
Tu es reviewer sénior. Tu n'as PAS écrit le plan — tu le lis froid.
Pour chaque section :
- Une hypothèse non écrite ?
- Un cas edge manquant ?
- Un risque sous-estimé ?
- Une étape qui pourrait casser une invariance existante ?

Rends `revised-plan.md` avec :
- Ce qui va bien (court, factuel)
- Les concerns priorisés (P0/P1/P2)
- Des propositions concrètes de changement
```

**Critère de sortie** : chaque concern P0 est soit intégré au plan, soit explicitement tranché avec raison. Pas de "on verra".

## Phase 3 — Implement

Une fois plan + revised-plan validés, **le plan est figé**. L'implementer ne décide plus — il exécute.

```markdown
---
name: ultra-implementer
description: Exécute un plan figé. N'invente pas. Si le plan est ambigu, s'arrête.
tools: Read, Edit, Write, Bash
model: sonnet
isolation: worktree
---
Tu appliques plan.md littéralement.
Si une étape n'est pas faisable en l'état : **stop**, reporte à l'utilisateur.
À la fin, lance les tests. Si rouge, commit uniquement si c'est attendu.
```

**Critère de sortie** : tests verts, plan intégralement suivi, aucune étape skippée en silence.

## Phase 4 — Judge

Un subagent indépendant relit le résultat sur grille fixe.

```markdown
---
name: ultra-judge
description: Juge le code livré par rapport au plan et au revised-plan. Score strict 0-10.
tools: Read, Grep, Bash
model: opus
---
Tu n'as PAS vu la conversation.
Compare code livré vs plan.md + revised-plan.md + tests.
Grille :
- Complétude (toutes étapes du plan faites)  — /3
- Conformité invariants projet (CLAUDE.md)   — /2
- Qualité tests                               — /2
- Lisibilité                                   — /2
- Risques résiduels                            — /1
Rends `verdict.md` avec score total + concerns.
```

**Critère de sortie** : score ≥ 7/10 **et** aucun P0 non résolu. Sinon : retour en implement ou plan selon la nature du concern.

## Budget tokens par phase

| Phase | Max turns | Modèle | Estim. tokens |
|-------|-----------|--------|--------------|
| Plan | 10 | Opus | 30-60k |
| Review | 6 | Opus | 20-40k |
| Implement | 25 | Sonnet | 60-150k |
| Judge | 6 | Opus | 20-40k |

Total typique : 150-300k tokens par cycle. Coût réel : $2-5. Comparé à 3h de rework humain, la rentabilité est immédiate.

## Hook Stop — automatiser la relance

Si le judge renvoie < 7, on boucle :

```json
{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "agent",
            "agent": "ultra-judge",
            "prompt": "Juge le travail de cette session."
          },
          {
            "type": "command",
            "command": ".claude/hooks/check-score.sh"
          }
        ]
      }
    ]
  }
}
```

Le script `check-score.sh` parse `verdict.md` ; si score < 7, exit 2 avec les concerns en stderr. Claude voit le retour et re-rentre en phase adéquate (souvent Implement).

**Cap dur** : un compteur d'itérations dans `.claude/memory/project_cycle.md`. Au-delà de 3 itérations, tu reprends la main — une boucle infinie signale un problème de plan, pas d'implémentation.

## Comparatif — one-shot vs cycle

| Aspect | One-shot | UltraPlan/UltraReview |
|--------|----------|----------------------|
| Durée totale | 30-60 min | 90-120 min |
| Tokens consommés | 80-150k | 150-300k |
| Coût API | $0.5-1.5 | $2-5 |
| Rework humain post-livraison | 2-4h | 0-30 min |
| Qualité code (score Judge) | 4-6/10 | 7-9/10 |
| Bugs produits | 3-5 | 0-1 |

**Point clé** : le cycle est **plus cher en tokens** mais **moins cher en temps humain**. Sur des tâches à fort enjeu (feature clé, refacto délicat), ça gagne largement.

## Quand NE PAS l'utiliser

- Tâche de 10 minutes (renommer une fonction, ajouter un log).
- Tâche explorée (prototype jetable).
- Débuggage d'incident — tu veux une réponse vite, pas un artefact qualité.

La règle : UltraPlan/UltraReview = pour du code **qui va vivre**. Pas pour du throwaway.

## Piège #1 — planner qui code

Un planner qui "montre" le code viole son rôle. Il doit **décrire** quoi faire, pas le faire. Sinon l'implementer devient décoratif. Contrainte explicite : `tools: Read, Grep, Glob` (pas d'Edit/Write).

## Piège #2 — reviewer complaisant

Un reviewer qui dit "plan très bien ✓" sans trouver d'axes d'amélioration n'a pas fait son job. Ton prompt doit lui imposer de **trouver au moins 3 concerns P0/P1** sur un plan non trivial. Si rien, le plan est probablement trop vague.

## Piège #3 — judge qui voit la conversation

Si le judge hérite du context de l'implementer, il a les mêmes biais. **Toujours** le lancer comme subagent **frais**, avec juste les artefacts (plan.md, verdict.md, code) en lecture.
