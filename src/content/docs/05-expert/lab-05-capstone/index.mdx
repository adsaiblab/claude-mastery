## Pourquoi un capstone freeform

Les labs précédents te tenaient la main : starter, tests, validate binaire.
La vraie compétence, c'est **scoper toi-même** une mission — choisir le bon
archétype, savoir quand t'arrêter, capitaliser le retour d'expérience.
Ce lab te force à le faire une fois proprement, avec des templates qui te
donnent le minimum de structure pour ne pas dériver.

## Choisir une mission (4 archétypes)

Tu choisis **un seul** des quatre :

1. **Audit & rapport** — tu lances un fan-out d'auditeurs (cf. [Lab 03 alt](/03-multi-agents/lab-03-agents/)) sur ton repo et tu produis un rapport priorisé. Idéal si ton repo a > 6 mois de dette technique.
2. **Refactor guidé** — tu choisis un module pénible (200-500 lignes), tu poses un cycle UltraPlan/UltraReview/Implement (cf. [Lab 05](/05-expert/lab-05/)), et tu compares avec ton instinct one-shot.
3. **Agent custom** — tu crées un subagent spécialisé pour une tâche que tu fais à la main 1×/semaine (review d'un type de PR, génération d'un type de test, etc.). Livraison = `.claude/agents/<nom>.md` + 3 cas d'usage validés.
4. **Routine automatisée** — tu poses une routine cron ou webhook (cf. [Lab 04 alt](/04-production/lab-04-routines/)) avec idempotence + DLQ sur un workflow récurrent réel (digest, sync, ETL).

**Règle** : si tu hésites entre deux, prends celui qui te débloquera **demain matin**, pas celui qui paraît le plus impressionnant.

## Démarrer

```bash
cd labs/lab-05-capstone
./setup.sh              # crée work/ avec les 4 templates Markdown
cd work/
```

`setup.sh` te crée quatre fichiers à remplir :

- `mission.md` — repo cible, archétype, critères de succès
- `plan.md` — étapes, budget temps, points de check
- `session-log.md` — journal de ce que tu as réellement fait (commandes, prompts, dérapages)
- `retrospective.md` — ce qui a marché, ce qui a foiré, ce que tu garderais

Le script vérifie :

1. Les 4 fichiers existent dans `work/` et sont **non-vides** (> 200 octets chacun).
2. `mission.md` contient les sections `## Repo cible`, `## Archétype`, `## Critères de succès`.
3. `plan.md` contient `## Étapes` avec au moins 4 items numérotés.
4. `retrospective.md` contient les 3 sections obligatoires.
5. `session-log.md` cite **au moins 2 prompts** (lignes commençant par `>` ou bloc ` ```prompt `).

Pas de check sur la correctness du code livré — c'est ton repo, tu juges.

- Bonus : refais le même lab dans 3 mois avec un autre archétype — tu verras la différence.

## Ce qu'on vérifie vraiment

1. Tu as **scopé toi-même** une mission qui te débloque concrètement.
2. Tu as **un livrable réel** (commit, PR, agent, routine) — pas juste de la doc.
3. Le `session-log.md` montre **les prompts qui ont marché** (et ceux qui ont dérapé).
4. La rétrospective contient **au moins une décision réutilisable** pour ton prochain capstone.
5. Tu as respecté ton budget temps initial (ou tu as documenté pourquoi tu l'as dépassé).
6. Les artefacts sont **suffisants** pour que tu reprennes ce travail froid dans 6 semaines.
