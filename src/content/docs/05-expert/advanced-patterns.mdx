## Pattern 1 — Doer/Judge

**Idée** : un agent fait, un autre note. Le second n'a pas vu la conversation du premier.

```
Doer (Sonnet) : implémente la tâche
   ↓ (artefact)
Judge (Opus) : grille de notation 0-10, indépendant
   ↓
score ≥ 7 ? → done : retour Doer avec concerns
```

**Quand** : tâche où la qualité est mesurable mais subjective (review code, tests, doc).

**Implémentation minimale** via hook Stop :

```json
{
  "hooks": {
    "Stop": [{
      "hooks": [
        { "type": "agent", "agent": "judge", "prompt": "Note ce qui a été fait." },
        { "type": "command", "command": ".claude/hooks/check-score.sh" }
      ]
    }]
  }
}
```

`check-score.sh` parse le verdict, exit 2 si `< 7`. Claude reprend la main.

**Piège classique** : Judge qui hérite du contexte du Doer = mêmes biais. Toujours `Agent()` frais.

→ [Fiche doer-judge](/architecture-patterns/doer-judge/)

## Pattern 2 — Fan-out / Fan-in

**Idée** : éclater une tâche sur N subagents en parallèle, agréger les résultats.

```
                       ┌──→ subagent A (module 1) ──┐
Main : split + dispatch├──→ subagent B (module 2) ──┼──→ Main : merge
                       └──→ subagent C (module 3) ──┘
```

**Quand** :
- Audit multi-modules (un audit par dir).
- Recherche de pattern dans un repo (un agent par chunk).
- Génération parallèle de variantes (3 propositions de copy).

**Critères de bonne tâche fan-out** :
1. Chaque branche est **indépendante** (pas d'effet bord cross-branche).
2. Le résultat de chaque branche est **borné** (≤ quelques k tokens).
3. La fusion est **déterministe** (concat, vote, max).

**Anti-pattern** : éclater une tâche dont les branches ont besoin de se voir → tu refais du séquentiel cher avec un détour.

```python
import asyncio
from claude_agent_sdk import query, ClaudeAgentOptions

async def audit_module(path):
    options = ClaudeAgentOptions(
        cwd=path,
        max_turns=5,
        system_prompt="Audit. Rends 5 lignes max.",
    )
    chunks = []
    async for msg in query(prompt="audit ./", options=options):
        chunks.append(str(msg))
    return "\n".join(chunks)

results = await asyncio.gather(
    audit_module("services/auth"),
    audit_module("services/billing"),
    audit_module("services/notifications"),
)
```

## Pattern 3 — Writer / Reviewer

**Idée** : un agent écrit, un autre relit avec critères explicites de passage.

Différent du Doer/Judge : ici on **collabore en N tours** plutôt que single-shot + verdict.

```
Writer : draft v1 → file
Reviewer : lit, annote, propose changements explicites
Writer : applique ou justifie un refus
Reviewer : valide ou itère
   ↓ (max 3 tours)
Done
```

**Quand** : production de contenu (doc, ADR, copy, README), refacto de prose, tests à formuler.

**Critères de passage à donner au Reviewer** :

```markdown
- Aucun jargon non défini au premier emploi.
- Chaque section ≤ 200 mots.
- Le titre est actionnable.
- Pas de phrases passives sauf raison expresse.
```

Plus les critères sont concrets, moins le Reviewer dérive vers du subjectif.

→ [Fiche writer-reviewer](/architecture-patterns/writer-reviewer/)

## Pattern 4 — Mémoire cross-session

**Idée** : ce qu'on a appris en session N doit être disponible en session N+1, sans tout recharger.

```
.claude/memory/
├── MEMORY.md                    ← index, lu auto à chaque session
├── project_<sujet>.md           ← faits du projet
├── feedback_<topic>.md          ← guidances utilisateur
├── reference_<system>.md        ← pointeurs externes
└── user_<topic>.md              ← profil utilisateur
```

**Règles** :
- `MEMORY.md` reste **petit** (< 200 lignes). Une ligne par mémoire avec un hook descriptif.
- Chaque mémoire dans son fichier typé avec frontmatter (`name`, `description`, `type`).
- Mise à jour **immédiate** quand un fait change. Pas de "je le ferai plus tard".

**Anti-pattern fréquent** : tout coller dans CLAUDE.md → 30k tokens chargés à chaque session. CLAUDE.md = règles structurantes. MEMORY.md = faits dynamiques.

→ [Fiche context-engineering](/architecture-patterns/context-engineering/)

## Pattern 5 — Cascade de modèles

**Idée** : tu n'as pas besoin d'Opus partout. Mappe la complexité de l'étape au modèle adapté.

```
Étape 1 : trier 200 PRs        → Haiku 4.5  (rapide, $0.001/PR)
Étape 2 : implémenter le fix   → Sonnet 4.6 (capacité, $0.015/req)
Étape 3 : juger qualité finale → Opus 4.7   (jugement, $0.075/req)
```

**Économie typique** sur un pipeline 100 items : 60-80% de coût vs full Opus, qualité finale équivalente sur la décision critique (qui reste sur Opus).

```python
from claude_agent_sdk import ClaudeAgentOptions

triage = ClaudeAgentOptions(model="claude-haiku-4-5-20251001", max_turns=2)
implement = ClaudeAgentOptions(model="claude-sonnet-4-6", max_turns=15)
judge = ClaudeAgentOptions(model="claude-opus-4-7", max_turns=4)
```

**Critère de choix** :
- **Haiku** : trier, classer, extraire un champ, répondre court factuel.
- **Sonnet** : raisonner, coder, tool use multi-étapes — la majorité du travail.
- **Opus** : décision critique, planification ambitieuse, jugement final.

**Piège** : descendre en gamme là où la qualité paye (planning, judge). L'économie sur ces étapes est minime, le coût d'erreur élevé.

## Combinaison réelle — pipeline triage + fix + review

Mise en œuvre des 5 patterns ensemble sur un cas concret : **traiter automatiquement les bug reports d'un projet OSS**.

```
1. Cron quotidien → liste 50 issues fraîches
2. Cascade : Haiku trie (bug réel ? doublon ? feature request ?)         ← P5
3. Fan-out : 3 subagents Sonnet implementent les 3 bugs P0 en parallèle  ← P2
4. Pour chaque fix : Writer (Sonnet) drafte la PR, Reviewer (Sonnet) lit ← P3
5. Judge (Opus) note la PR finale ; < 7 → retour writer                  ← P1
6. progress.md mis à jour ; concerns persistent dans MEMORY.md           ← P4
```

Coût empirique sur 50 issues : ~$8-12 vs ~$45 en full Opus single-pass. Qualité PR équivalente (mesurée sur taux de merge humain post-review).

## Anti-pattern combinatoire — "tout, partout, tout le temps"

Empiler les patterns sans nécessité multiplie le coût et l'opacité :

- Doer/Judge **et** Writer/Reviewer **et** Fan-out sur la même tâche = 4-5x le coût pour un gain marginal.
- Cascade Haiku→Sonnet→Opus alors qu'une seule étape suffit = perte de fluidité.

**Règle** : commence avec **un seul** pattern, mesure, ajoute le second seulement si nécessaire.
