  Tu as 5 mécanismes en main : **subagent, agent team, skill, MCP, hook, plugin**.
  Toutes les briques se ressemblent dans leur promesse — "ça étend Claude". Mais

## La grille — quelle brique pour quoi

| Besoin | Bonne brique | Pourquoi |
|--------|-------------|----------|
| Déléguer une recherche isolée qui rend une synthèse | **Subagent** | Context isolé, prompt autoportant |
| Faire plusieurs streams de travail en parallèle | **Agent team** | Vraies sessions persistantes |
| Charger du savoir seulement quand pertinent | **Skill** | À la demande, peu coûteux en context |
| Connecter un système externe comme un tool | **MCP** | Tools typés, pas de parsing stdout |
| Exécuter une règle déterministe sur un événement | **Hook** | Code, pas prompt — toujours appliqué |
| Packager plusieurs de ces briques + versionner | **Plugin** | Distribution, update, sécurité |

**Règle mnémotechnique** :

- Question ouverte → **subagent**
- Parallélisme humain-visible → **team**
- Savoir conditionnel → **skill**
- Système externe → **MCP**
- Garde-fou sur événement → **hook**
- Partage d'équipe → **plugin**

## Arbre de décision

```
Je veux …
│
├─ … exécuter du code automatiquement à un moment précis
│    → Hook (PreToolUse / PostToolUse / SessionStart / Stop)
│
├─ … connecter un système externe (GitHub, DB, Slack…)
│    → MCP server
│
├─ … donner à Claude du savoir conditionnel
│    ├─ Utilisé tout le temps → CLAUDE.md (pas un skill)
│    └─ Utilisé parfois       → Skill
│
├─ … déléguer une sous-tâche bornée
│    → Subagent
│
├─ … faire travailler plusieurs streams en parallèle avec vrais terminaux
│    → Agent team
│
└─ … distribuer un ensemble à mon équipe / à la communauté
     → Plugin qui bundle le tout
```

**Contexte** : ton équipe veut qu'à chaque PR ouverte, un agent fasse une review automatique, commente les problèmes, et approuve si tout va bien.

**Contraintes** :

- Headless (GitHub Actions, pas d'humain).
- Doit créer un commentaire de review sur GitHub.

**Contexte** : tu veux refactorer 80 fichiers pour migrer d'une lib à une autre. Le repo entier fait 250k tokens — impossible à tenir en un context.

**Contraintes** :

- Tu veux garder une vue d'ensemble de la migration.
- Chaque fichier doit être traité de façon cohérente avec le plan.

## Les pièges d'architecture les plus courants

### Piège A — mettre en hook ce qui devrait être skill

Un hook qui formate du code est légitime. Un hook qui injecte **des instructions** à chaque session devient une règle que personne ne peut désactiver contextuellement. Un skill se charge seulement quand pertinent.

### Piège B — MCP pour tout

Chaque MCP server activé coûte du context (spec des tools). Active seulement ceux nécessaires au projet. Pour un repo doc-only, pas besoin de MCP GitHub.

### Piège C — subagent quand on voulait skill

Un subagent = nouveau Claude lancé. Coûteux. Si tu veux juste "ajouter du savoir à l'agent principal", un skill suffit. Subagent = pour **faire** quelque chose, pas pour **savoir**.

### Piège D — pas de plugin quand ça devient 5+ briques partagées

Tu as 3 agents + 2 hooks + un MCP config utilisés par 4 devs. Continuer à tout copier à la main = divergence en 2 semaines. Plugin = unification.
