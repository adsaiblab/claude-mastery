<Situation>
  Trois tâches indépendantes à finir avant 18h : migrer un endpoint, écrire les
  tests d'un module, et rédiger une RFC. Un seul Claude qui fait tout en série,
  c'est 3× plus long et ton context sature. Un **agent team** = plusieurs
  sessions Claude Code qui tournent **en parallèle**, chacune dans son tmux pane,
  coordonnées par une task list partagée.
</Situation>

## Concept — subagent vs agent team

| Dimension | Subagent | Agent team |
|-----------|----------|------------|
| Qui est aux commandes | L'agent principal | Toi, humain |
| Contexte | Isolé, éphémère | Session Claude Code complète, persistante |
| Durée de vie | Le temps d'un message | Heures / jours |
| Parallélisme | Lancés par le parent | Vrais terminaux parallèles |
| Communication | Un prompt → un résultat | Bidirectionnel via `SendMessage` |
| UI | Invisible (background) | Un pane par agent, tu vois tout |

**Règle de choix** : subagent pour une sous-tâche bornée ; team pour plusieurs **streams de travail** qui vivent en parallèle.

## Activation

Feature flag (2026) — encore experimental :

```bash
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
```

Depuis la CLI, la commande maîtresse :

```
/team <teammate-name>     — ajoute un coéquipier au team courant
/team list                — affiche les coéquipiers actifs
/cleanup-team             — ferme proprement les sessions du team
```

## Display modes

Quand tu ajoutes un teammate, tu choisis **comment tu veux le voir** :

| Mode | Comportement | Usage typique |
|------|-------------|---------------|
| **In-process** | Le teammate tourne dans la même fenêtre, navigation `Shift+↑/↓` entre vues | Teams légers (2-3 agents) |
| **tmux split** | Chaque teammate = un pane tmux | Teams de 3-5 agents sur un écran |
| **iTerm2 tabs** | Nouvel onglet par teammate | Mac, tu navigues à la souris |
| **Detached** | Le teammate tourne en arrière-plan, tu pull ses updates | Agents longs qui travaillent pendant que tu fais autre chose |

## Réutiliser les fichiers subagents

**Bonne nouvelle** : un fichier `.claude/agents/<name>.md` fait office de **définition de teammate** aussi. Tu n'as pas deux formats à maintenir.

```
.claude/agents/
├── implementer.md    ← utilisable comme subagent ET comme teammate
├── reviewer.md
└── docs-writer.md
```

La différence est au moment de l'invocation : `Agent({subagent_type: "implementer"})` pour un subagent, `/team implementer` pour un teammate.

## SendMessage — la communication entre teammates

Une fois plusieurs teammates actifs, tu peux leur parler entre eux via l'outil `SendMessage` :

```
User (dans main) : Demande à `reviewer` de regarder le diff de `implementer`.

Main :
  → SendMessage(to="reviewer",
      message="Implementer a fini auth/token.ts:45-120. Peux-tu reviewer ?")
  
Reviewer (dans son pane) :
  ← reçoit le message
  ← lit le code
  ← réponds via SendMessage(to="main", message="2 concerns : ...")
```

C'est ce qui distingue un team d'une simple collection de terminaux séparés : ils **peuvent** se parler quand tu leur dis.

## Task list partagée

Une task list vit au niveau du team, pas par agent :

```
[team: refacto-auth]
├─ [✓] planner   : rédiger plan d'extraction TokenService
├─ [⟳] implementer: appliquer plan (en cours, 7/15)
├─ [ ] tester    : écrire tests pour TokenService
└─ [ ] docs      : mettre à jour README.md
```

Tout le monde voit l'état. Celui qui finit une tâche passe au suivant, ou prend la main d'une bloquée.

## Pattern 1 — Doer + Judge en parallèle

Le Doer implémente. Le Judge, en permanence, regarde les edits au fur et à mesure et remonte des concerns :

```
Terminal 1 (doer)    │  Terminal 2 (judge)
────────────────────┼─────────────────────
Edit auth.ts         │
Edit auth.test.ts    │  ← "auth.ts:47 — gestion d'erreur absente"
Edit middleware.ts   │  ← "middleware.ts:12 — import circulaire potentiel"
```

Le doer ne s'arrête pas — le judge alimente une file de concerns que tu traites en fin de session.

## Pattern 2 — 3 explorations parallèles

Tu explores trois patterns d'archi différents pour une même décision :

```
Pane 1 : explore "event sourcing" approach
Pane 2 : explore "state machine" approach
Pane 3 : explore "transactional script" approach
```

Chacun bosse dans son fork de session. À la fin, tu synthétises les 3 propositions dans un terminal maître.

## Pattern 3 — long task + ping

Un agent tourne pendant 20 minutes sur une migration batch. Au lieu de le surveiller, tu lui demandes de t'envoyer un `SendMessage` quand il a fini **ou** qu'il est bloqué. Tu fais autre chose entre-temps.

```bash
claude --name batch-migration-042026 \
       --team-ping-on-stop \
       --prompt "Migre tous les fichiers du dossier /legacy vers /v2. Ping-moi quand fini."
```

## Piège #1 — trop d'agents

3 agents = productif. 8 agents = chaos. Tu perds plus de temps à les coordonner qu'à bosser. **Règle** : commence à 2, ajoute seulement quand tu sens un vrai goulet d'étranglement.

## Piège #2 — pas de cleanup

Un team laissé ouvert = N sessions Claude Code qui consomment des tokens idle si un hook `prompt` les réveille. **Toujours** `/cleanup-team` quand tu fermes la journée.

## Piège #3 — même repo, modifs concurrentes

Deux teammates qui éditent les mêmes fichiers = conflits à résoudre à la main. Si tes teammates écrivent, donne à chacun **son worktree** :

```
---
name: implementer
isolation: worktree
---
```

## Démo — une matinée efficace

```bash
# 9h00 — ouvrir le team
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
claude --name team-refacto-matinee
/team implementer
/team tester
/team docs-writer

# 9h01 — répartir
"implementer: applique plan.md étape 1 à 7"
"tester: commence à écrire les tests pour TokenService d'après plan.md §Tests"
"docs-writer: mets à jour README.md section auth"

# 9h30 — Shift+↓ pour voir où en est implementer
# 10h00 — SendMessage(to=implementer, "push la première PR quand t'as les tests verts")
# 12h00 — /cleanup-team ; 3 PRs posées au lieu d'une
```
