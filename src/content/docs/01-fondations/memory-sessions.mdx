## Deux mécanismes de persistance

Ne confonds pas les deux :

|             | **CLAUDE.md**    | **MEMORY.md + fichiers typés**                           |
| ----------- | ---------------- | -------------------------------------------------------- |
| Qui écrit ? | Toi              | Claude                                                   |
| Quand ?     | Avant la session | Pendant la session, quand il détecte une info clé        |
| Scope       | Projet ou user   | Par projet (par défaut)                                  |
| Objectif    | Règles stables   | Apprentissage cross-session                              |
| Format      | Libre, court     | Index + fichiers typés (user/feedback/project/reference) |

## Auto-memory — comment Claude se souvient

Quand tu dis : _"Note que les PRs doivent passer par la queue prioritaire `ci-fast`"_, Claude peut :

1. Créer ou mettre à jour `.claude/memory/project_ci.md` avec la règle.
2. Ajouter une ligne dans `.claude/memory/MEMORY.md` qui pointe vers ce fichier.

Au démarrage de la prochaine session, Claude scanne `MEMORY.md`, identifie les mémoires pertinentes à ta tâche courante, et charge les fichiers correspondants dans son contexte.

### Structure type

```
.claude/
└── memory/
    ├── MEMORY.md                         # index, ~1 ligne par fichier
    ├── user_profile.md                   # qui tu es, préférences
    ├── project_ci.md                     # processus CI/CD du projet
    ├── feedback_prefers_short_pr.md      # comportement correct à répéter
    └── reference_linear_project.md       # où tracker les bugs
```

### Format d'un fichier de mémoire

```markdown
---
name: Profil utilisateur
description: Rôle et préférences de l'utilisateur
type: user
---

Senior backend engineer, 8 ans Node, découvre React pour la première fois.
Préfère les explications brèves sauf si le problème est non-obvious.
Ne documente pas en prose longue — bullet points courts.
```

### Limites

- **200 lignes ou 25 KB max par fichier.** Au-delà, Claude peut tronquer.
- **Scope par worktree** par défaut. Un git worktree = un `.claude/memory/` séparé.
- Peut être désactivée via settings.

## La commande `/memory`

Pendant une session :

```
/memory                 — affiche ce qui est chargé
/memory edit            — ouvre l'index MEMORY.md dans ton éditeur
/memory clear           — décharge les mémoires du contexte courant
```

Utile pour **voir ce que Claude a appris sur toi/le projet** et pour nettoyer les mémoires obsolètes.

## Sessions — les bases

Claude Code stocke chaque session dans `~/.claude/projects/<repo-path>/<session-id>.jsonl`. Chaque ligne = un event (user message, tool call, etc.).

### Lancer une session

```bash
# Nouvelle session anonyme
claude

# Nouvelle session nommée (essentiel pour la retrouver)
claude --name refacto-auth

# Continuer la dernière session de ce repo
claude -c
# ou
claude --continue
```

### Reprendre une session précise

```bash
# Picker interactif — choisis dans une liste
/resume
# ou :
claude --resume
```

### Fork — dupliquer à un état donné

```bash
claude --fork-session <session-id>
```

Crée une copie pour explorer une autre piste sans toucher à l'originale. Utile pour :

- "Je veux essayer une autre approche sans perdre l'état actuel."
- "Je vais tester une hypothèse risquée, avec fallback."

## Sessions avancées — `/branch`, tags

```
/branch <nom>           — crée une branche à partir du tour courant
/rename <nouveau-nom>   — renomme la session
/tag <tag>              — ajoute un tag (pour recherche)
```

Les tags et le nom sont recherchables via le picker `/resume`.

## Piège classique — deux terminaux, même session

Si tu lances `claude --resume <same-id>` dans **deux terminaux** simultanément, les events des deux sessions **s'entrelacent dans le même fichier JSONL**. Conséquence : la reprise ultérieure voit une conversation chaotique et bizarre.

**Règle de sécurité :** un session-id = un terminal à la fois. Si tu veux vraiment deux fils parallèles, `--fork-session` pour créer un second ID.

## Naming — comment bien nommer

Convention utile :

```
<action>-<domaine>-<date?>
```

Exemples :

- `refacto-auth-042026`
- `investigate-prod-incident-checkout`
- `draft-rfc-multi-agents`

Ça te permet de retrouver en une seconde dans le picker, même après deux semaines.

## Démo — un lundi matin après une pause

```bash
# Vendredi soir
claude --name refacto-auth-042026
# ... tu bosses, /compact, tu fermes ...

# Lundi matin
cd ~/projects/api
claude --resume
# → picker affiche : "refacto-auth-042026 (vendredi, 14 tours)"
# tu sélectionnes, tu continues.
```

Sans `--name`, tu aurais eu "session-2026-04-25-14-48" et tu serais perdu.
