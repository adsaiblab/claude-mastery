Le **frontmatter** est le bloc YAML en tête des fichiers Markdown définissant
des subagents, skills, slash-commands et plugins. C'est là que tu déclares
le comportement (modèle, tools, permissions) ; le body Markdown est le
prompt système. Cette page liste tous les champs acceptés par type.

## Subagent (`.claude/agents/<name>.md`)

| Champ            | Type   | Usage                                                                                                      |
| ---------------- | ------ | ---------------------------------------------------------------------------------------------------------- |
| `name`           | string | Identifiant unique (kebab-case). Utilisé par `Agent(name=...)`.                                            |
| `description`    | string | Une ligne. Sert au routage automatique : Claude lit cette description pour décider d'invoquer le subagent. |
| `model`          | enum   | `opus` / `sonnet` / `haiku` ou ID complet. Hérite du parent si omis.                                       |
| `tools`          | list   | Tools accessibles. `Read, Edit, Write, Bash, Grep, Glob, WebFetch, ...`. Restreint = plus sûr.             |
| `permissionMode` | enum   | `default` / `acceptEdits` / `plan` / `bypassPermissions`.                                                  |
| `maxTurns`       | int    | Hard cap sur les tours du subagent. Évite les boucles.                                                     |
| `skills`         | list   | Skills auto-chargés au démarrage de ce subagent.                                                           |
| `mcpServers`     | list   | Serveurs MCP exposés à ce subagent (sous-ensemble du parent).                                              |
| `hooks`          | object | Hooks scopés à ce subagent.                                                                                |
| `memory`         | bool   | Lit `MEMORY.md` au démarrage (défaut `true`).                                                              |
| `background`     | bool   | Démarre en background (Agent Teams).                                                                       |
| `effort`         | enum   | `low` / `medium` / `high` — adapte le thinking.                                                            |
| `isolation`      | enum   | `worktree` crée un git worktree dédié pour ce subagent.                                                    |
| `color`          | string | Couleur dans l'UI Agent Teams.                                                                             |
| `initialPrompt`  | string | Préfixé à chaque invocation.                                                                               |

### Exemples par cas d'usage

**Subagent read-only (auditor) :**

```yaml
---
name: auditor
description: Use to audit a single file or service. Read-only. Produces audit report.
model: haiku
tools: Read, Grep, Glob
maxTurns: 5
memory: false
---
```

`memory: false` — l'auditor n'a pas besoin de l'historique persistant ; il
travaille sur un input frais à chaque invocation.

**Subagent qui peut écrire mais isolé (agent feature-dev) :**

```yaml
---
name: feature-dev
description: Implement a feature in isolation. Has write access via worktree.
model: sonnet
tools: Read, Edit, Write, Bash(git:*), Bash(npm:*)
isolation: worktree
maxTurns: 30
effort: medium
---
```

`isolation: worktree` crée un git worktree dédié — l'agent peut commit
librement sans toucher la branche principale.

**Subagent qui orchestre (planner) :**

```yaml
---
name: planner
description: Use first to break a feature request into actionable steps.
model: opus
tools: Read, Grep, Glob
permissionMode: plan
maxTurns: 10
effort: high
initialPrompt: "Tu produis des plans, pas du code. Sortie en Markdown structuré."
---
```

`permissionMode: plan` force le mode plan — le planner ne peut pas écrire,
même si on lui en donnait les tools.

## Skill (`.claude/skills/<name>/SKILL.md`)

| Champ           | Type   | Usage                                                                                 |
| --------------- | ------ | ------------------------------------------------------------------------------------- |
| `name`          | string | Identifiant unique.                                                                   |
| `description`   | string | Décrit **quand** invoquer le skill — Claude charge à la demande sur match sémantique. |
| `allowed-tools` | list   | Tools utilisables pendant l'exécution du skill.                                       |
| `model`         | enum   | Force un modèle (rare — généralement on hérite).                                      |

Le corps du SKILL.md contient les instructions ; les fichiers adjacents (`scripts/`, `templates/`) sont chargés à la demande.

### Anatomie d'un skill

```
.claude/skills/refactor-react/
├── SKILL.md                  # frontmatter + instructions principales
├── scripts/
│   ├── extract-component.mjs # script utilitaire
│   └── inline-component.mjs
└── templates/
    └── component.tsx.tmpl    # template référencé par le skill
```

Le skill peut déclarer ses scripts dans `allowed-tools` :

```yaml
---
name: refactor-react
description: Use when extracting/inlining React components, especially when refactoring a large file.
allowed-tools: Read, Edit, Write, Bash(node:scripts/*)
---
```

## Slash-command (`.claude/commands/<name>.md`)

| Champ           | Type   | Usage                                           |
| --------------- | ------ | ----------------------------------------------- |
| `description`   | string | Affiché dans l'UI quand l'utilisateur tape `/`. |
| `allowed-tools` | list   | Tools autorisés pendant la commande.            |
| `model`         | enum   | Modèle forcé pour la commande.                  |
| `argument-hint` | string | Indication d'arguments (ex: `<filename>`).      |

Le corps du `.md` est le **prompt** envoyé à Claude quand la commande est invoquée. `$ARGUMENTS` est interpolé avec les args utilisateur. Voir [slash-commands](/reference/slash-commands/) pour le détail des variables et des exemples complets.

## Plugin (`.claude-plugin/plugin.json`)

| Champ         | Type   | Usage                              |
| ------------- | ------ | ---------------------------------- |
| `name`        | string | Identifiant unique du plugin.      |
| `version`     | semver | `1.2.3`.                           |
| `description` | string | Une ligne.                         |
| `author`      | object | `{ "name": "...", "url": "..." }`. |
| `repository`  | string | URL git.                           |
| `license`     | string | SPDX (`MIT`, `Apache-2.0`, ...).   |
| `keywords`    | list   | Tags pour la marketplace.          |

Le contenu du plugin (subagents, skills, hooks, commands) vit dans les sous-dossiers conventionnels (`agents/`, `skills/`, `commands/`, `hooks/`).

### Stratégie keywords pour la marketplace

Les `keywords` sont indexés pour la recherche dans la marketplace plugins.
Bonnes pratiques :

- 3-7 keywords (au-delà, dilution).
- Mélanger **technologie** (`react`, `python`, `terraform`) et **type d'usage** (`audit`, `refactor`, `testing`).
- Éviter les keywords trop génériques (`utility`, `helper`) — pas découvrables.
- Inclure les noms des outils ou frameworks externes intégrés (`playwright`, `kubernetes`, `linear`).

## Comparaison frontmatter — subagent vs skill vs command

| Dimension            | Subagent                      | Skill                             | Slash-command         |
| -------------------- | ----------------------------- | --------------------------------- | --------------------- |
| Déclaration          | `agents/<name>.md`            | `skills/<name>/SKILL.md`          | `commands/<name>.md`  |
| Restriction tools    | `tools:`                      | `allowed-tools:`                  | `allowed-tools:`      |
| Restriction modèle   | `model:`                      | `model:`                          | `model:`              |
| Argument utilisateur | (via `Agent(name=, prompt=)`) | (sémantique, pas d'arg explicite) | `$ARGUMENTS` / `$1`   |
| Routage              | description sémantique        | description sémantique            | nom de la commande    |
| Mémoire              | `memory: bool`                | (héritée)                         | (héritée)             |
| Hooks scopés         | `hooks:`                      | non                               | non                   |
| Worktree dédié       | `isolation: worktree`         | non                               | non                   |
| Cas d'usage typique  | Tâche complète, autonome      | Procédure réutilisable            | Raccourci utilisateur |

Règle de choix :

- **Slash-command** si l'utilisateur invoque explicitement (`/audit src/foo.ts`).
- **Skill** si tu veux que Claude charge automatiquement une procédure quand le contexte le demande.
- **Subagent** si tu veux une fenêtre séparée (context isolé, modèle dédié, parallélisme).

## Pièges classiques

| Piège                                                                 | Conséquence                                                              | Fix                                                              |
| --------------------------------------------------------------------- | ------------------------------------------------------------------------ | ---------------------------------------------------------------- |
| Description vague (« Outil utile »)                                   | Routage inefficace : Claude n'invoque jamais ou trop souvent             | Décrire **quand** invoquer (« Use when X happens »).             |
| Tools redondants (`Edit, Write` pour un agent qui ne fait que `Edit`) | Surface d'attaque inutile                                                | Restreindre au minimum.                                          |
| `model:` non déclaré                                                  | Hérite du parent — comportement non déterministe selon la session source | Forcer `model:` pour des subagents critiques (auditor, judge).   |
| `maxTurns` absent sur un agent récursif                               | Risque de boucle (agent qui ré-invoque ses pairs)                        | `maxTurns: 5-10` pour les agents bornés.                         |
| `skills:` qui charge un skill volumineux dans chaque invocation       | Coût tokens à chaque appel                                               | Réserver aux skills < 1 KB. Pour les gros, charger à la demande. |
| Plugin sans `version` ou avec `0.0.1` figé                            | Pas de signal de maturité aux utilisateurs marketplace                   | Bumper la version à chaque change ; utiliser semver strict.      |
| Body trop long dans une slash-command (> 100 lignes)                  | Tokens consommés à chaque invocation                                     | Externaliser en script Bash ou en skill.                         |

## Conventions

- **Description** : décrit **quand** plutôt que **quoi**. C'est l'élément clé du routage.
- **Tools** : restreindre au minimum nécessaire. Un subagent qui n'écrit jamais ne devrait pas avoir `Edit/Write`.
- **Model** : Opus pour planifier/juger, Sonnet pour faire, Haiku pour trier.
- **Frontmatter en haut, prose en dessous** : pas de prose avant le `---` d'ouverture (sinon le YAML n'est pas parsé).
- **Versionner les frontmatter** dans le repo : c'est le contrat équipe pour les agents.
